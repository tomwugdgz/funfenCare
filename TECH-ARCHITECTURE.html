🏗️ 架构概览

芬芬陪护采用五端分离架构，通过AI分析实现一对一个性化陪护，解决传统摄像头侵犯隐私、手表佩戴不适的核心痛点。

核心设计理念：
  • ✅ 无摄像头：WiFi体感替代摄像头，不拍摄画面，保护隐私
  • ✅ 无穿戴设备：无需老人佩戴手表/手环，消除抵触心理
  • ✅ AI个性化：每个用户独立建模，AI学习生活习惯，提供专属陪护方案
  • ✅ 五端协同：独居用户、监护人、后台服务、全球AI监控、上门服务五端联动
端 用户角色 核心功能 技术形态
端1 独居用户 一键SOS、日常打卡、接受陪护 App适老化版 / 微信小程序

数据库设计核心表

-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY,
    phone VARCHAR(20) UNIQUE NOT NULL,
    role VARCHAR(20) CHECK (role IN ('elder', 'guardian', 'caregiver', 'admin')),
    name VARCHAR(50),
    created_at TIMESTAMP DEFAULT NOW()
);

-- 亲情关系表
CREATE TABLE family_relations (
    elder_id UUID REFERENCES users(id),
    guardian_id UUID REFERENCES users(id),
    relationship VARCHAR(20),  -- 子女/配偶/其他
    PRIMARY KEY (elder_id, guardian_id)
);

-- WiFi设备表
CREATE TABLE wifi_devices (
    id UUID PRIMARY KEY,
    elder_id UUID REFERENCES users(id),
    device_token VARCHAR(64) UNIQUE,
    status VARCHAR(20) DEFAULT 'online',
    last_seen TIMESTAMP
);

-- CSI活动数据表（时序）
CREATE TABLE csi_activity (
    id BIGSERIAL PRIMARY KEY,
    device_id UUID REFERENCES wifi_devices(id),
    timestamp TIMESTAMP NOT NULL,
    activity_score FLOAT,        -- 0-1活动强度
    anomaly_score FLOAT,         -- 异常分数
    raw_data JSONB               -- 原始CSI数据
);

-- 服务订单表
CREATE TABLE service_orders (
    id UUID PRIMARY KEY,
    elder_id UUID REFERENCES users(id),
    guardian_id UUID REFERENCES users(id),
    caregiver_id UUID REFERENCES users(id),
    service_type VARCHAR(20),    -- door_check/chat/companion
    status VARCHAR(20),
    scheduled_at TIMESTAMP,
    completed_at TIMESTAMP,
    feedback TEXT
);

-- AI用户画像表
CREATE TABLE ai_user_profiles (
    user_id UUID PRIMARY KEY REFERENCES users(id),
    wake_time_avg TIME,          -- 平均起床时间
    sleep_time_avg TIME,         -- 平均入睡时间
    activity_pattern JSONB,      -- 活动模式
    interests VARCHAR[],         -- 兴趣标签
    health_notes TEXT,           -- 健康备注
    last_updated TIMESTAMP
);

🌐 端4: Web4.0 MCP 全球AI监控端

🤖
Web4.0 MCP 全球AI监控端 (脱敏)
全球独居老人AI监控面板，脱敏数据驱动的AI分析

定位与价值

Web4.0 MCP理念：
  • MCP = Model Context Protocol：标准化AI模型接口，全球AI可接入分析
  • 脱敏数据：所有数据经过匿名化处理，仅保留活动模式，不含身份信息
  • 全球视野：聚合全球独居老人活动数据，发现共性模式，优化AI模型
  • AI自治：AI自动识别异常模式、推荐陪护方案、预测健康风险

脱敏数据规范

原始数据 脱敏处理 保留信息
用户ID 哈希加密 + 加盐 唯一标识（不可逆）
手机号 完全移除 无
地址 模糊到城市级别 城市、区域
年龄 分段处理 年龄段（60-70/70-80/80+）
活动数据 保留原始数值 活动强度、时间分布
健康备注 NLP关键词提取 关键词标签（如"高血压"）

AI监控面板功能

🌍

全球热力图

展示全球各城市独居老人活动热力分布，识别高风险区域

📊

模式分析

AI识别常见活动模式（规律型/不规律型/孤僻型等），分类统计

🔮

风险预测

基于历史数据预测健康风险，提前预警（如活动量持续下降）

💡

陪护推荐

AI推荐最佳陪护方案（如"适合戏曲聊天""需要增加户外活动"）

接单群可以采用微信群 + 平台App双模式运行：

查看微信群模式
  1. 每个社区/街道建立专属微信群
  2. 群成员：网格员、志愿者、平台调度员
  3. 调度员在群内发布任务："XX小区X栋X户，需要上午9点上门打卡"
  4. 服务人员回复接单，前往现场
  5. 完成后在群内打卡（照片+文字反馈）
  6. 平台自动同步打卡记录到子女端

优点：即时通信、操作简单、适合临时调度

缺点：难以追踪、不易统计、依赖人工

AI陪护推荐引擎

# AI陪护推荐逻辑
def generate_care_recommendation(user_profile, current_data):
    """
    根据用户画像和当前状态，生成个性化陪护建议
    """
    recommendations = []
    
    # 1. 活动量分析
    if current_data.activity_score < user_profile.activity_pattern.get('threshold', 0.3):
        recommendations.append({
            "type": "increase_activity",
            "suggestion": "今日活动量偏低，建议安排轻度陪聊",
            "urgency": "medium",
            "action": "schedule_chat",
            "preferred_topics": user_profile.interests[:2]
        })
    
    # 2. 作息异常检测
    if abs(current_data.wake_time - user_profile.daily_rhythm.wake_time) > timedelta(hours=2):
        recommendations.append({
            "type": "schedule_anomaly",
            "suggestion": f"今日起床时间异常（{current_data.wake_time}），建议确认健康状况",
            "urgency": "high",
            "action": "door_check"
        })
    
    # 3. 社交需求评估
    if days_since_last_interaction(user_profile) > 3:
        recommendations.append({
            "type": "social_need",
            "suggestion": "已超过3天无人交流，建议安排陪聊",
            "urgency": "medium",
            "action": "schedule_chat",
            "chat_type": "voice_call"  # 优先语音
        })
    
    return recommendations

🔒 隐私保护方案

为什么不用摄像头？

摄像头方案的问题：
  • 侵犯隐私：老人在家中被24小时监控，心理不适
  • 数据泄露风险：视频流被黑客截获
  • 老人抵触：很多老人拒绝安装摄像头
  • 法律合规：需要明确告知并获取同意

WiFi体感方案的优势

WiFi体感方案：
  • ✅ 不采集图像：仅检测WiFi信号变化，无画面无声音
  • ✅ 无需穿戴：老人无需佩戴任何设备
  • ✅ 全天候监控：不受光线、遮挡影响
  • ✅ 成本低廉：ESP32硬件成本不到100元
  • ✅ 易于部署：即插即用，老人无需操作

📡 WiFi体感硬件方案

技术原理

WiFi CSI（Channel State Information）技术通过分析WiFi信号在传播过程中的微小变化，检测人体活动。当人在WiFi信号覆盖范围内移动时，会改变信号的传播路径，导致CSI值变化。AI算法通过分析这些变化，可以识别：

  • 是否有人：检测到CSI变化即有人活动
  • 活动强度：CSI变化幅度反映活动强度
  • 活动区域：多天线的CSI差异可定位活动区域
  • 跌倒检测：特定CSI模式可识别跌倒动作

核心数据流详解

数据流1: 设备→后台→监护人
  1. ESP32设备采集CSI信号，每秒10次
  2. 边缘计算提取活动特征，压缩数据
  3. 通过MQTT协议上报到IoT网关
  4. IoT网关转发到后台服务
  5. 后台进行AI分析，生成活动分数
  6. 监护人端通过WebSocket实时接收更新
  7. 监护人App可视化展示活动曲线
数据流2: 监护人下单→服务人员→反馈
  1. 监护人在App选择服务类型、时间
  2. 创建订单，推送到接单群/服务人员端
  3. 服务人员接单，获取地址和备注
  4. 上门后GPS打卡，拍照上传
  5. 填写服务记录（老人状态、备注）
  6. 提交反馈，自动通知监护人
  7. 监护人查看反馈，评价服务
数据流3: AI模型训练→全球优化
  1. 本地AI模型分析用户活动数据
  2. 生成模型梯度更新（不包含原始数据）
  3. 上传梯度到中央服务器
  4. 联邦学习聚合全球梯度
  5. 更新全局模型参数
  6. 下发到各本地节点
  7. 本地模型迭代优化
